# DeskLink 테스트 가이드 (초보자용, 아주 상세)

이 문서는 컴퓨터/개발을 전혀 모르는 사람도 따라 할 수 있도록 테스트 방법을 한 단계씩 설명한다.
모르는 단어가 나오면 맨 아래 [부록 A: 용어 사전](#부록-a-용어-사전) 을 먼저 보면 된다.

---

## 0. 이 문서를 읽는 법

- 회색 상자 안의 글자는 직접 입력(복사·붙여넣기)하는 명령어다. 예:
  ```bash
  python3 --version
  ```
- 명령어는 보통 터미널(검은 화면)에 붙여넣고 Enter(엔터)를 누르면 실행된다.
- 각 단계마다 "이렇게 나오면 성공" / "이렇게 나오면 실패" 예시를 넣었다.
- 테스트는 쉬운 것부터 어려운 것 순서(레벨 1 → 4)로 되어 있다. 위에서부터 차례로 하면 된다.

---

## 1. 테스트 4단계 한눈에 보기

| 레벨 | 무엇을 테스트? | 기기 필요? | 난이도 | 걸리는 시간 |
|------|----------------|-----------|--------|------------|
| 1 | 통신 규격(숫자·바이트)이 맞는지 | 불필요 (컴퓨터만) | 아주 쉬움 | 1분 |
| 2 | 맥(Mac) 프로그램 내부 로직 | 불필요 | 보통 | 5~10분 |
| 3 | 안드로이드 앱 내부 로직 | 불필요 | 보통 | 5~15분 |
| 4 | 실제로 맥+태블릿 연결해서 동작 | 필요 (맥+태블릿) | 어려움 | 30분+ |

> 핵심: 레벨 1~3은 기기(태블릿) 없이 컴퓨터에서만 확인하는 "자동 테스트"다. 코드가 규칙대로 작동하는지 프로그램이 스스로 검사한다. 레벨 4는 사람이 직접 눈으로 보는 테스트다.

## 2. 준비물 (처음 한 번만)

테스트하려면 아래 프로그램이 컴퓨터에 있어야 한다. 없으면 설치가 필요하다. (설치가 부담되면 레벨 1만 먼저 해봐도 된다. 레벨 1은 거의 모든 맥에 기본으로 있는 Python만 있으면 된다.)

| 레벨 | 필요한 것 | 확인 명령어 | 없을 때 설치 |
|------|-----------|-------------|--------------|
| 1 | Python 3 | `python3 --version` | 맥에 기본 포함(없으면 안내창이 뜸) |
| 2 | Xcode (맥용) | 앱 목록에 "Xcode" | App Store에서 "Xcode" 검색·설치 |
| 3 | Android Studio + JDK 17 | 앱 목록에 "Android Studio" | [developer.android.com/studio](https://developer.android.com/studio) |
| 4 | 위 전부 + `adb` + 태블릿 | `adb --version` | `brew install android-platform-tools` |

설치가 필요하면 `docs/BUILD_AND_TEST.md`의 "1. 사전 요구사항"을 함께 참고한다.

## 3. 터미널 여는 법 (맥 기준) — 레벨 1·2·3 공통

"터미널"은 명령어를 입력하는 검은 창이다.

1. 키보드에서 `Command(⌘)` + `Space(스페이스바)` 를 동시에 누른다. → 화면 가운데 검색창(Spotlight)이 뜬다.
2. `터미널` 또는 `Terminal` 이라고 입력한다.
3. Enter(엔터)를 누른다. → 검은(또는 흰) 창이 열린다. 이게 터미널이다.

### 프로젝트 폴더로 이동하기 (매번 필요)

터미널이 열리면, 우리 프로젝트 폴더로 "들어가야" 한다. 아래를 그대로 복사해서 붙여넣고 Enter:

```bash
cd /Users/gr/00.workspaces/99_study/android-multi-display
```

- `cd` 는 "폴더 이동(change directory)"이라는 뜻이다.
- 잘 들어갔는지 확인하려면 아래를 입력한다. 파일 목록이 보이면 성공이다.
  ```bash
  ls
  ```
  성공 예시(이런 이름들이 보임):
  ```
  README.md   android   docs   macos   tools   IMPLEMENTATION_PLAN.md ...
  ```
  실패 예시: `No such file or directory` → 경로를 잘못 붙여넣은 것. 3번 처음부터 다시.

> 아래 모든 명령어는 이 폴더 안에서 실행한다고 가정한다. 터미널을 새로 열면 매번 위 `cd` 명령을 먼저 실행해야 한다.

## 4. 레벨 1 — 통신 규격 테스트 (제일 쉬움, 1분)

이게 무슨 테스트? 맥과 태블릿은 정해진 "약속된 숫자 형식"으로 대화한다. 이 형식이 규격서(`docs/protocol-spec.md`)와 정확히 맞는지 컴퓨터가 자동으로 확인해 준다. 기기 없이 컴퓨터만으로 1분이면 끝난다.

### 실행 방법

터미널에 아래를 붙여넣고 Enter:

```bash
python3 tools/protocol_vectors.py
```

### 결과 읽는 법

성공: 맨 아래 줄에 이렇게 나오면 통과다.
```
ALL CHECKS PASS
```
화면에는 이런 표도 함께 나온다(정상):
```
TOUCH_EVENT(20B)         023F0000003E800000800001000462D53C8ABAC0
FRAMED_TOUCH             0000001520023F0000003E800000800001000462D53C8ABAC0
...
touch RT: ... -> OK
ALL CHECKS PASS
```

실패: 어딘가에 `MISMATCH` 또는 `FAILURE(S)` 라는 글자가 보이면 규격이 어긋난 것이다.
```
   !! MISMATCH expected 023F...
1 FAILURE(S)
```
→ 이 경우 그 화면 전체를 캡처해서 개발자(또는 저에게) 전달하면 된다.

`python3: command not found` 가 나오면 Python이 없는 것이다. 처음엔 맥이 "개발자 도구를 설치할까요?" 창을 띄운다. 설치를 누르고 끝난 뒤 다시 실행한다.

> 이 테스트가 통과하면, 맥과 안드로이드가 "같은 언어로 대화한다"는 가장 중요한 부분이 검증된 것이다.

## 5. 레벨 2 — 맥(Mac) 프로그램 자동 테스트 (5~10분)

이게 무슨 테스트? 맥 서버 프로그램의 내부 부품들(화면 압축, 데이터 포장, 터치 변환 등)이 각각 제대로 동작하는지 자동으로 검사한다. 태블릿은 필요 없다. Xcode가 설치되어 있어야 한다.

두 가지 방법이 있다. 초보자는 방법 A(클릭)를 추천한다.

### 방법 A — Xcode에서 클릭으로 (쉬움)

1. 터미널에서(3번의 `cd` 이후) 아래를 입력해 Xcode로 프로젝트를 연다:
   ```bash
   open -a Xcode macos/DeskLink/Package.swift
   ```
   → Xcode가 열리고, 잠시 "패키지 준비 중(Resolving/Preparing)" 표시가 나온다. 다 끝날 때까지 1~3분 기다린다.
   - 참고: `-a Xcode` 없이 `open ...Package.swift`만 하면, `.swift`의 기본 프로그램(예: VS Code, Google Antigravity 등)이 대신 열릴 수 있다. `-a Xcode`는 "반드시 Xcode로 열라"는 뜻이다.
   - `Unable to find application named 'Xcode'` 라고 나오면 Xcode가 설치되지 않은 것 → App Store에서 "Xcode" 설치.
2. 상단 메뉴에서 Product(프로덕트) → Test(테스트)를 클릭한다. (단축키: `Command(⌘) + U`)
3. 왼쪽/하단에 테스트가 하나씩 실행되며 결과가 표시된다.

성공: 각 테스트 이름 옆에 초록색 체크 표시가 뜬다. 상단에 "Test Succeeded"라고 나온다.

실패: 빨간색 X 표시가 뜬다. 그 테스트를 클릭하면 어디가 틀렸는지 설명이 나온다. 화면을 캡처해 전달하면 된다.

> 빌드 에러가 먼저 날 수 있음: 이 프로젝트의 최근 코드는 아직 실제 컴파일(빌드) 검증 전이다. 테스트가 실행되기 전에 빨간 에러가 뜨면, 그 에러 메시지를 캡처해서 전달해 달라. 그걸 보고 코드를 고치면 된다.

### 방법 B — 터미널로 (Xcode를 열지 않고)

터미널에 아래를 붙여넣고 Enter:
```bash
cd macos/DeskLink
swift test
```
(끝나면 원래 폴더로 돌아가려면 `cd ..` 를 입력)

성공 예시(마지막쯤에):
```
Test Suite 'All tests' passed at ...
	 Executed 40 tests, with 0 failures
```

실패 예시:
```
Test Suite 'All tests' failed ...
	 Executed 40 tests, with 2 failures
```
→ 위쪽에 어떤 테스트가 왜 실패했는지 나온다. 캡처해서 전달.

### 특정 테스트 하나만 실행하고 싶을 때 (선택)
```bash
cd macos/DeskLink
swift test --filter EncodedFrameTests
```

## 6. 레벨 3 — 안드로이드 앱 자동 테스트 (5~15분)

이게 무슨 테스트? 안드로이드 앱의 내부 부품들(데이터 포장/해체, 연결 관리, 영상 조각 재조립 등)을 자동 검사한다. 태블릿은 필요 없다. Android Studio + JDK 17이 필요하다.

### 방법 A — Android Studio에서 클릭으로 (쉬움)

1. Android Studio 앱을 연다.
2. 첫 화면에서 Open(열기) 클릭 → 폴더 선택 창에서 아래 폴더를 찾아 선택:
   ```
   /Users/gr/00.workspaces/99_study/android-multi-display/android
   ```
   (반드시 끝이 `android` 인 폴더를 선택한다.)
3. 열리면 하단에 "Gradle Sync(그레이들 동기화)"가 자동으로 돈다. 끝날 때까지 기다린다(처음엔 인터넷으로 부품을 받느라 5~15분 걸릴 수 있다). 하단 진행 막대가 사라지면 완료.
4. 왼쪽 Project(프로젝트) 패널에서 폴더를 펼친다:
   `app` → `src` → `test` → `java` → `com.desklink.android`
5. `com.desklink.android`(또는 그 안의 `data`) 폴더를 마우스 오른쪽 클릭 → Run 'Tests in ...'(테스트 실행) 클릭.
6. 하단에 결과가 나온다.

성공: 테스트 이름들 옆에 초록 체크, 하단에 "Tests passed: N".

실패: 빨간 X. 클릭하면 상세 이유가 보인다. 캡처해서 전달.

### 방법 B — 터미널로

터미널에 아래를 붙여넣고 Enter (처음엔 Gradle을 받느라 오래 걸림):
```bash
cd android
./gradlew test
```

성공 예시(마지막 줄):
```
BUILD SUCCESSFUL in 2m 13s
```

실패 예시:
```
BUILD FAILED in 1m 40s
...
> There were failing tests. See the report at: .../index.html
```

### 결과를 예쁘게 보기 (HTML 리포트)
터미널로 실행했다면, 아래 파일을 더블클릭하면 브라우저에서 보기 좋은 결과표가 열린다:
```
android/app/build/reports/tests/testDebugUnitTest/index.html
```
초록 = 통과, 빨강 = 실패.

> 처음 실행 시 주의: `./gradlew` 는 첫 실행에서 Gradle과 라이브러리를 인터넷으로 내려받는다. 시간이 오래 걸리고, 인터넷이 필요하다. 두 번째부터는 빨라진다.

## 7. 레벨 4 — 실제 기기 연결 테스트 (사람이 눈으로 확인, 30분+)

이게 무슨 테스트? 맥과 태블릿을 실제로 연결해서, 태블릿이 맥의 확장 모니터처럼 동작하는지 직접 본다. 가장 중요하지만 준비가 많다. 먼저 레벨 1~3이 통과한 뒤에 하는 것을 권장한다.

### 7.1 준비 (한 번만)

1. 태블릿에서 개발자 옵션을 켠다: 설정 → 휴대전화 정보 → 빌드번호를 7번 연속 탭 → "개발자가 되었습니다" 문구.
2. 설정 → 개발자 옵션 → USB 디버깅을 켠다.
3. USB 케이블로 태블릿과 맥을 연결한다.
4. 태블릿 화면에 "이 컴퓨터에서 USB 디버깅을 허용하시겠습니까?" 창이 뜨면 허용/항상 허용을 누른다.
5. 맥 터미널에서 연결 확인:
   ```bash
   adb devices
   ```
   성공 예시:
   ```
   List of devices attached
   R52T1234ABC     device
   ```
   `command not found` → adb 미설치. `brew install android-platform-tools` 실행.
   목록이 비어 있음 → 케이블/USB 디버깅 허용 다시 확인.

### 7.2 맥 서버 실행 + 권한 부여 (아주 중요)

1. Xcode로 맥 앱을 실행한다(레벨 2 방법 A의 1번처럼 `Package.swift`를 열고 실행 버튼 또는 `Command(⌘) + R`). → 화면 위쪽 메뉴 막대에 DeskLink 아이콘(모니터 두 개 모양)이 생긴다.
2. 처음 실행하면 두 가지 권한을 반드시 켜야 한다:
   - 화면 기록 권한: 시스템 설정 → 개인정보 보호 및 보안 → 화면 기록 → 목록에서 DeskLink를 켠다(체크). (화면을 복제하려면 필요)
   - 손쉬운 사용 권한: 시스템 설정 → 개인정보 보호 및 보안 → 손쉬운 사용 → DeskLink를 켠다. (태블릿 터치를 맥 클릭으로 넣으려면 필요)
3. 권한을 켠 뒤 앱을 껐다 켠다(권한은 재시작해야 적용되는 경우가 많다).

> 권한을 안 켜면: 화면이 안 나오거나(화면 기록), 터치를 해도 맥이 반응하지 않는다(손쉬운 사용).

### 7.3 연결 통로 만들기 (adb reverse)

맥이 "서버", 태블릿이 "손님"이다. 태블릿이 USB를 통해 맥에 접속하도록 통로 3개를 연다. 맥 서버 앱이 자동으로 열어주지만, 수동으로 확인/설정하려면 터미널에 입력:
```bash
adb reverse tcp:7100 tcp:7100
adb reverse tcp:7101 tcp:7101
adb reverse tcp:7102 tcp:7102
adb reverse --list
```
`--list` 결과에 7100, 7101, 7102 세 줄이 보이면 통로가 열린 것이다.

> 참고: 반드시 `adb reverse` 다(`adb forward` 아님). 방향이 반대면 연결되지 않는다.

### 7.4 태블릿 앱 실행 및 연결

1. 태블릿에서 DeskLink 앱을 연다.
2. 연결 화면에서 주소를 `127.0.0.1`(= localhost)로 둔다.
3. 연결(Connect)을 누른다.
4. 몇 초 안에 태블릿에 맥의 확장 화면이 나타나면 성공이다.

### 7.5 눈으로 확인하는 시나리오 (하나씩 체크)

아래는 `IMPLEMENTATION_PLAN.md` §5의 시나리오를 초보자용으로 풀어 쓴 것이다. 각 항목을 해보고 되면 체크한다.

- [ ] 연결: 태블릿에 맥 화면이 뜬다.
- [ ] 화면 갱신: 맥에서 창을 움직이면 태블릿에서도 부드럽게 따라 움직인다(끊김이 심하지 않음).
- [ ] 터치: 태블릿의 한 지점을 누르면, 맥의 확장 화면 같은 위치가 클릭된다.
- [ ] 드래그: 태블릿에서 손가락을 끌면 맥에서 드래그가 된다.
- [ ] 연결 끊김 감지: USB를 뽑으면(또는 Wi-Fi 끊으면) 잠시 뒤 앱이 "연결 끊김" 상태로 바뀐다.
- [ ] 정상 종료: 태블릿에서 뒤로가기를 하면 깔끔히 종료되고 맥 화면이 사라진다.

각 항목이 안 되면, 어느 항목이 어떻게 안 됐는지(예: "터치는 되는데 위치가 안 맞음") 적어서 전달하면 원인을 좁힐 수 있다.

### 7.6 Wi-Fi로 하고 싶다면 (선택)
USB 대신 같은 와이파이에서 하려면: adb 통로가 필요 없고, 태블릿 앱의 주소에 맥의 IP 주소를 입력한다. (맥 IP는 시스템 설정 → 네트워크에서 확인) USB보다 약간 느릴 수 있다.

## 8. 자주 나오는 문제와 해결 (초보자용)

| 상황 | 이렇게 해보세요 |
|------|----------------|
| `command not found: python3` | 맥이 띄우는 "개발자 도구 설치" 창에서 설치를 누른 뒤 다시 실행 |
| `command not found: adb` | 터미널에 `brew install android-platform-tools` 입력(Homebrew 필요) |
| `cd ... No such file or directory` | 경로를 잘못 붙여넣음. 3번의 `cd` 명령을 정확히 다시 복사 |
| `adb devices`에 기기가 안 보임 | USB 케이블 교체, 태블릿에서 USB 디버깅 허용 재확인, `adb kill-server` 후 `adb devices` |
| Xcode/Studio에서 빨간 에러(빌드 실패) | 최근 코드가 아직 컴파일 검증 전이라 그럴 수 있음. 에러 메시지 전체를 캡처해 전달 |
| Gradle이 계속 도는 중 | 처음엔 부품을 내려받느라 오래 걸림. 인터넷 연결 확인하고 기다리기 |
| 태블릿에 화면이 안 나옴 | 맥의 화면 기록 권한 켰는지 확인 → 앱 재시작 |
| 터치해도 맥이 반응 없음 | 맥의 손쉬운 사용 권한 켰는지 확인 → 앱 재시작 |
| 연결 자체가 안 됨(USB) | `adb reverse --list`로 7100~7102 통로 확인, 없으면 7.3 다시 |

> 어떤 문제든 화면을 캡처해서 전달하면 훨씬 빨리 해결할 수 있다. 특히 빨간 글씨(에러)는 그 내용이 원인을 알려준다.

## 9. 추천 진행 순서 (요약 체크리스트)

- [ ] 1단계: `python3 tools/protocol_vectors.py` → `ALL CHECKS PASS` 확인 (1분)
- [ ] 2단계: Xcode에서 `Command(⌘) + U`로 맥 테스트 → 초록 체크 확인
- [ ] 3단계: Android Studio에서 test 폴더 우클릭 → Run → 초록 체크 확인
- [ ] 4단계: 태블릿 연결 → 권한 부여 → 확장 화면·터치 눈으로 확인

1~3단계가 모두 초록이면 "코드 로직"은 검증된 것이고, 4단계까지 되면 "실제 제품"이 동작하는 것이다.

## 부록 A: 용어 사전

- 터미널(Terminal): 명령어를 글자로 입력해 컴퓨터를 다루는 검은 창.
- 명령어(command): 터미널에 입력하는 한 줄 지시. 붙여넣고 Enter를 누르면 실행된다.
- `cd`: "폴더 이동". 특정 폴더 안으로 들어가는 명령.
- `ls`: 현재 폴더 안의 파일/폴더 목록 보기.
- 빌드(build): 사람이 쓴 코드를 컴퓨터가 실행할 수 있는 형태로 조립하는 과정. "컴파일"과 비슷한 뜻.
- 단위 테스트(unit test): 프로그램의 작은 부품 하나하나가 잘 작동하는지 자동으로 검사하는 코드.
- Xcode: 애플이 만든, 맥/아이폰 앱을 개발·테스트하는 프로그램.
- Android Studio: 구글이 만든, 안드로이드 앱을 개발·테스트하는 프로그램.
- Gradle / gradlew: 안드로이드 앱을 빌드·테스트하는 도구. `./gradlew`는 그 도구를 실행하는 명령.
- adb: "Android Debug Bridge". 맥과 안드로이드 기기를 USB로 연결해 명령을 주고받는 도구.
- adb reverse: 안드로이드 기기가 USB를 통해 맥의 서버에 접속하도록 통로를 만드는 명령.
- 포트(port) 7100/7101/7102: 맥 서버가 각각 제어/영상/터치 통신에 사용하는 "문 번호".
- 골든 벡터(golden vector): "정답 바이트 값". 통신 형식이 정답과 같은지 비교하는 기준.
- 호스트(host) / 클라이언트(client): 서버(맥) / 손님(태블릿). 여기서 서버는 맥, 클라이언트는 태블릿이다.
- 권한(permission): 앱이 화면 녹화·입력 제어 같은 민감한 기능을 쓰도록 사용자가 허락하는 것.
